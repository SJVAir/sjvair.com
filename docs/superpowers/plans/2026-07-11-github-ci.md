# GitHub Actions CI + Branch Protection Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Run `pytest` automatically on every push/PR via GitHub Actions, and block merges into `main` until that check passes.

**Architecture:** A single-job workflow (`.github/workflows/test.yml`) reuses the project's existing `docker compose --profile test` setup — the same `db`/`test` services and healthchecks documented in CLAUDE.md — pre-building the `test` target image with Docker Buildx + GitHub Actions cache for speed. A classic branch protection rule on `main`, applied via `gh api`, requires that job's `test` status check to pass before merging, with `enforce_admins: false` so admins can still push directly in an emergency.

**Tech Stack:** GitHub Actions (`actions/checkout`, `docker/setup-buildx-action`, `docker/build-push-action`), Docker Compose (existing `docker-compose.yml`), `gh` CLI for branch protection.

## Global Constraints

- Repo: `SJVAir/sjvair.com`, default branch `main`.
- No secrets used — all `.env.test` values written into the workflow are the same non-sensitive dummy/test-only values already in the local gitignored `.env.test` (confirmed via `git ls-files`: `.env.test` is not tracked).
- Tests only need Postgres (`db` service) — `camp.settings.test` runs Huey in immediate/synchronous mode and uses locmem cache, so `queue`/`cache` services are not required for the test run (see `camp/settings/test.py`).
- Required status check context name must be exactly `test` (the job name), matching what branch protection will require.
- Never use `enforce_admins: true` — admins must retain emergency direct-push ability per approved design.

- [ ] **Step 1: Add an explicit image tag to the `test` service in `docker-compose.yml`**

Edit `docker-compose.yml` so the `test` service reads:

```yaml
  test:
    <<: *build-base
    build:
      context: .
      target: test
    image: sjvair-com-test:latest
    command: pytest
    env_file:
      - .env.test
    depends_on:
      db:
        condition: service_healthy
    profiles: ["test"]
```

(only the `image: sjvair-com-test:latest` line is new, inserted after the `build:` block).

- [ ] **Step 2: Verify local tests still pass unaffected by the compose change**

Run: `docker compose run --rm test pytest -x -q`
Expected: test suite runs and passes exactly as before (the `image:` line only names the build output; it doesn't change behavior).

- [ ] **Step 3: Create the workflow file**

Create `.github/workflows/test.yml`:

```yaml
name: Test

on:
  push:
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Write .env.test
        run: |
          cat > .env.test <<'EOF'
          ALLOWED_HOSTS=localhost
          DATABASE_URL=postgres://sjvair:changeme@db:5432/sjvair
          REDIS_URL=redis://redis:6379
          MEMCACHED=memcached:11211
          DJANGO_DEBUG=0
          DJANGO_SETTINGS_MODULE=camp.settings.test
          SECRET_KEY=ci-test-secret-key-not-for-production-use
          PURPLEAIR_READ_KEY=ci-test-purpleair-read-key
          PURPLEAIR_WRITE_KEY=ci-test-purpleair-write-key
          PURPLEAIR_GROUP_ID=1400
          EOF

      - name: Build test image
        uses: docker/build-push-action@v6
        with:
          context: .
          target: test
          tags: sjvair-com-test:latest
          load: true
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Run tests
        run: docker compose --profile test run --rm test pytest

      - name: Tear down
        if: always()
        run: docker compose --profile test down -v
```

- [ ] **Step 4: Commit**

```bash
git add docker-compose.yml .github/workflows/test.yml
git commit -m "ci: add GitHub Actions workflow to run pytest on push/PR"
```

- [ ] **Step 5: Push a branch and confirm the workflow runs**


Expected: a workflow run named "Test" starts, the `test` job completes successfully, `gh run watch` exits 0. If it fails, read the step logs with `gh run view --log-failed` and fix before proceeding — do not move to Task 2 with a red workflow.

### Task 3: Configure branch protection on `main`

**Files:** none (GitHub repo configuration via API)

**Interfaces:**
- Consumes: the `test` job name produced in Task 1.

- [ ] **Step 1: Apply the branch protection rule**

```bash
gh api -X PUT repos/SJVAir/sjvair.com/branches/main/protection \
  -H "Accept: application/vnd.github+json" \
  -F 'required_status_checks[strict]=false' \
  -f 'required_status_checks[contexts][]=test' \
  -F 'enforce_admins=false' \
  -F 'required_pull_request_reviews=null' \
  -F 'restrictions=null'
```

- [ ] **Step 2: Verify the rule is active**

```bash
gh api repos/SJVAir/sjvair.com/branches/main/protection --jq '{contexts: .required_status_checks.contexts, strict: .required_status_checks.strict, enforce_admins: .enforce_admins.enabled}'
```

Expected output:

```json
{"contexts": ["test"], "strict": false, "enforce_admins": false}
```
